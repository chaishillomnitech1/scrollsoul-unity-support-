## üöÄ How to Contribute

### 1. Fork and Clone

```bash
# Fork the repository on GitHub, then clone your fork
git clone https://github.com/YOUR_USERNAME/scrollsoul-unity-support-.git
cd scrollsoul-unity-support-

# Add upstream remote
git remote add upstream https://github.com/chaishillomnitech1/scrollsoul-unity-support-.git
```

### 2. Create a Branch

Create a branch with a descriptive name that reflects your contribution:

```bash
git checkout -b feature/amazing-contribution
# or
git checkout -b fix/issue-description
# or
git checkout -b docs/documentation-improvement
```

Branch naming conventions:
- `feature/` - New features or enhancements
- `fix/` - Bug fixes
- `docs/` - Documentation updates
- `refactor/` - Code refactoring
- `test/` - Test improvements

### 3. Make Your Changes

- Write clear, concise commit messages that reflect the LOVE principle
- Keep commits focused and atomic
- Follow existing code style and conventions
- Add tests for new functionality
- Update documentation as needed

### 4. Commit Guidelines

Follow this format for commit messages:

```
<type>: <short summary>

<detailed description of changes>

Aligns with: <LOVE|UNITY|TRUTH|ETERNAL>
```

Example:
```
feat: Add automated deployment workflow

Implements CI/CD pipeline with GitHub Actions for automated
testing and deployment. Includes security scanning and compliance checks.

Aligns with: UNITY, TRUTH, ETERNAL
```

### 5. Push and Create Pull Request

```bash
# Push your changes
git push origin feature/amazing-contribution

# Create a Pull Request on GitHub
```

## üìã Pull Request Process

1. **Ensure CI Passes**: All automated checks must pass
2. **Update Documentation**: Include relevant documentation updates
3. **Add Tests**: Include tests that validate your changes
4. **Describe Changes**: Provide a clear description of what and why
5. **Link Issues**: Reference any related issues

### Pull Request Template

Your PR description should include:

```markdown
## Description
[Clear description of changes]

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Documentation update
- [ ] Refactoring

## Alignment with Core Principles
- [ ] LOVE: Compassionate and empathetic implementation
- [ ] UNITY: Harmonious integration with existing code
- [ ] TRUTH: Transparent and well-documented
- [ ] ETERNAL: Sustainable and maintainable

## Testing
- [ ] Existing tests pass
- [ ] New tests added and passing
- [ ] Manual testing completed

## Checklist
- [ ] Code follows project style guidelines
- [ ] Documentation updated
- [ ] No breaking changes (or clearly documented)
- [ ] Commits are signed and verified
```

## üîç Code Review Process

## üêõ Reporting Issues

When reporting issues:

1. **Search First**: Check if the issue already exists
2. **Be Specific**: Provide clear, reproducible steps
3. **Include Context**: OS, versions, environment details
4. **Be Respectful**: Remember the LOVE principle

### Issue Template

```markdown
## Description
[Clear description of the issue]

## Steps to Reproduce
1. [First step]
2. [Second step]
3. [...]

## Expected Behavior
[What should happen]

## Actual Behavior
[What actually happens]

## Environment
- OS: [e.g., Ubuntu 22.04]
- Version: [e.g., 1.0.0]
- Additional context: [any other relevant info]
```
